    **1. Create src/lib/mesh/solid.ts:**

    **`buildSolidMesh(terrainGeometry: THREE.BufferGeometry, basePlateThicknessMM: number): THREE.BufferGeometry`**

    Takes the terrain surface geometry (from buildTerrainGeometry) and closes it into a watertight solid by adding:

    a) **Base plate** — a flat rectangle at z = -basePlateThicknessMM, same XY footprint as the terrain surface bounding box. The base is composed of two triangles forming a rectangle.

    b) **Side walls** — four vertical walls connecting the terrain perimeter edges to the base plate edges. For each edge of the terrain surface (north, south, east, west perimeter), create triangles that drop vertically down from each perimeter vertex to z = -basePlateThicknessMM, then connect to the base.

    Implementation approach:
    1. Extract the terrain surface positions from the geometry's position attribute.
    2. Identify perimeter vertices (those on the boundary of the grid — first row, last row, first column, last column). These are the vertices where x or y is at the min/max extent.
    3. For the base plate: create a rectangle at z = -basePlateThicknessMM using the min/max X and min/max Y from the terrain positions.
    4. For side walls: walk each perimeter edge. For two consecutive perimeter vertices (p1, p2) on the terrain surface, their corresponding base points are (p1.x, p1.y, -basePlateThicknessMM) and (p2.x, p2.y, -basePlateThicknessMM). Create 2 triangles forming a quad between the terrain edge and base edge.
    5. **CRITICAL for manifold-ness:** All faces must have consistent winding (counter-clockwise when viewed from outside). Terrain surface normals point up (+Z), base plate normal points down (-Z), side walls point outward. Get winding correct.
    6. Merge all triangles (terrain surface + base + walls) into a single BufferGeometry using `THREE.BufferGeometryUtils.mergeGeometries()` or by manually concatenating the position arrays.
    7. Compute vertex normals: `geometry.computeVertexNormals()`

    **Important detail:** The terrain mesh from martini has indexed geometry (shared vertices). The perimeter vertices are those that lie on x=0, x=widthMM, y=0, or y=depthMM (after the centering offset — adjust for the centering applied in terrain.ts). You need to find these vertices by checking their positions, not by index.

    Actually, a simpler and more robust approach for the perimeter:
    - The terrain mesh positions after centering span from -widthMM/2 to +widthMM/2 in X and -depthMM/2 to +depthMM/2 in Y.
    - The base plate rectangle corners are at (-widthMM/2, -depthMM/2, -basePlateThicknessMM) and (+widthMM/2, +depthMM/2, -basePlateThicknessMM).
    - For side walls, instead of walking terrain perimeter edges (complex with indexed RTIN mesh), use a simpler approach: create four walls as rectangular grids that span the full edge, with the top edge Z values sampled from the terrain. Since martini produces an adaptive mesh, the terrain edge might have varying vertex density. A practical approach:
      1. Sample the terrain Z values along each edge at regular intervals (e.g., 64 samples per edge).
      2. For each pair of adjacent samples, create a quad (2 triangles) from the terrain edge point down to the base.
    - This produces clean walls that seal the mesh.

    **2. Create src/lib/export/validate.ts:**

    **`getManifold(): Promise<ManifoldModule>`** — Lazy-load manifold-3d WASM:
    ```typescript
    import Module from 'manifold-3d';
    let manifoldLib: Awaited<ReturnType<typeof Module>> | null = null;

    export async function getManifold() {
      if (!manifoldLib) {
        manifoldLib = await Module();
        manifoldLib.setup();
      }
      return manifoldLib;
    }
    ```

    **`validateMesh(geometry: THREE.BufferGeometry): Promise<{ isManifold: boolean; error?: string; triangleCount: number }>`**
    1. Extract non-indexed positions from geometry (call `geometry.toNonIndexed()` first if indexed)
    2. Get positions as Float32Array
    3. Create a Manifold mesh from the vertex data
    4. Check if it's valid. If manifold-3d throws or returns error status, `isManifold = false` with error message.
    5. Return result with triangle count = positions.length / 9

    If manifold-3d proves difficult to initialize or has API issues at runtime, fall back to a simple boundary-edge check: a mesh is manifold if every edge is shared by exactly 2 triangles. Implement this as a fallback:
    - Build edge map: for each triangle, add 3 edges (sorted vertex pair → count)
    - Non-manifold if any edge has count != 2
    This is less thorough but catches the most common issues.

    **3. Create src/lib/export/stlExport.ts:**

    **`exportToSTL(mesh: THREE.Mesh): { buffer: ArrayBuffer; sizeBytes: number; triangleCount: number }`**
    1. Import `STLExporter` from `'three/addons/exporters/STLExporter.js'` (NOT from 'three/examples/jsm/...' — use the addons path).
    2. `const exporter = new STLExporter()`
    3. `const buffer = exporter.parse(mesh, { binary: true }) as ArrayBuffer`
    4. Compute triangleCount = (buffer.byteLength - 84) / 50 (binary STL: 84-byte header + 50 bytes per triangle)
    5. Return { buffer, sizeBytes: buffer.byteLength, triangleCount }

    **`downloadSTL(buffer: ArrayBuffer, filename: string): void`**
    1. `const blob = new Blob([buffer], { type: 'application/octet-stream' })`
    2. Create `<a>` element, set href to `URL.createObjectURL(blob)`, set download to filename
    3. Click it, then `URL.revokeObjectURL(url)`

    **`generateFilename(bbox: BoundingBox): string`** — Auto-generate filename from location:
    1. Compute center lat/lon from bbox
    2. Use reverse geocoding if available, otherwise fallback to `terrain-{lat}-{lon}.stl`
    3. Slugify: lowercase, replace spaces with hyphens, append '-terrain.stl'
    4. For simplicity (no reverse geocoding API call), just use coordinate-based name: `terrain-{centerLat.toFixed(2)}-{centerLon.toFixed(2)}.stl`
    - Actually, per user decision, the filename should be location-based like "mount-rainier-terrain.stl". Since we don't have a reverse geocoding API wired up yet, use the geocoding search text if available in the store. Add an optional `locationName: string | null` field to the store (set when geocoding search selects a result). If locationName exists, slugify it + "-terrain.stl". Otherwise, fall back to coordinate-based.
