**Root cause context:** The bug is a Y-axis inversion in `buildTerrainGeometry` (terrain.ts). Tile coordinates have Y increasing downward (north = low Y). The stitched elevation array has row 0 = northernmost data. But `buildTerrainGeometry` maps `vy=0` to `y = -depthMM/2` (the SOUTH end of the mesh), placing northern elevation data at the southern mesh position. For multi-tile grids, this flips the terrain vertically, producing the "4 rotated quadrants" effect.

**Test approach — synthetic elevation data with known spatial pattern:**

1. Create a synthetic `ElevationData` object with gridSize=257 (single Martini-compatible grid) that has a **spatially asymmetric** elevation pattern. Specifically:
   - NW quadrant (low vx, low vy in array = row 0, col 0): elevation = 100m (high — simulates a mountain in the NW)
   - NE quadrant (high vx, low vy): elevation = 50m
   - SW quadrant (low vx, high vy): elevation = 25m
   - SE quadrant (high vx, high vy): elevation = 10m (low — simulates a valley in the SE)
   - Use smooth gradients within each quadrant to avoid Martini artifacts

2. Call `buildTerrainGeometry(elevationData, params)` with widthMM=150, depthMM=150, geographicWidthM=1000, geographicDepthM=1000, exaggeration=1.5, minHeightMM=5, maxError=5.

3. Read back the positions from the resulting BufferGeometry.

4. **Assert spatial correctness:**
   - Sample vertices near the four corners of the mesh.
   - The vertex at approximately `(x = -widthMM/2, y = +depthMM/2)` (top-left = NW in a Z-up coordinate system) should have the HIGHEST elevation (Z value), since the NW quadrant has elevation=100m.
   - The vertex at approximately `(x = +widthMM/2, y = -depthMM/2)` (bottom-right = SE) should have the LOWEST elevation, since the SE quadrant has elevation=10m.

   - The NE corner Z should be between NW and SW.

  <action>
Fix the Y-axis mapping in two functions in `src/lib/mesh/terrain.ts`:

**1. `buildTerrainGeometry` — line 115 (the Y position calculation):**

Current (WRONG):
```typescript
const y = (vy / (gridSize - 1)) * depthMM - depthMM / 2;
```

This maps vy=0 (north in tile/array space) to y=-depthMM/2 (south in mesh space). INVERT it:

Fixed:
```typescript
const y = (1 - vy / (gridSize - 1)) * depthMM - depthMM / 2;
```

Now vy=0 (north) maps to y=+depthMM/2 (north in mesh space), and vy=gridSize-1 (south) maps to y=-depthMM/2 (south in mesh space). Geographic north = positive Y.

**2. `updateTerrainElevation` — line 182 (the reverse grid index recovery):**

Current (WRONG — matches the old forward mapping):
```typescript
const vy = Math.round((y + depthMM / 2) / depthMM * (gridSize - 1));
```

This must be updated to invert the recovery to match the new forward mapping:

Fixed:
```typescript
const vy = Math.round((1 - (y + depthMM / 2) / depthMM) * (gridSize - 1));
```

This correctly recovers vy from y by inverting the formula `y = (1 - vy/(gridSize-1)) * depthMM - depthMM/2`.

**DO NOT** modify stitch.ts or tiles.ts. The stitching logic is correct — the bug is purely in the coordinate mapping from grid space to mesh space. The stitched array correctly has north at row 0 (matching tile conventions). The fix is to make the mesh builder respect that convention by mapping row 0 to positive Y.

**DO NOT** change the X mapping — it is correct (left = negative X, right = positive X).

After applying both fixes, run ALL tests (not just the new one) to ensure no regressions. The existing 32 tests must still pass alongside the new spatial arrangement test.
