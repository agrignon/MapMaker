<objective>
Prevent UI freezes during mesh generation and export, and add dense-area OOM safeguards.

Purpose: FNDN-03 requires mesh generation to run in a Web Worker so the UI never freezes. The preview path already uses `meshBuilder.worker.ts` for roads and buildings. The export pipeline (`ExportPanel.tsx`) currently runs `buildAllBuildings` and `buildRoadGeometry` synchronously on the main thread, which can freeze the browser for 1-3 seconds on dense urban areas. This plan routes the export pipeline's building/road geometry steps through the existing worker and adds a bbox area cap to prevent OOM crashes on very large selections.

Output: Non-blocking export pipeline, bbox area cap (soft warning at 4 km2, hard cap at 25 km2).
</objective>

Move the heaviest export pipeline steps (building and road geometry generation) off the main thread by reusing the existing worker infrastructure. This prevents UI freezes during STL export on dense urban areas.

**Key constraint (from STATE.md):** Merge all geometry into three typed arrays before postMessage. Never send per-feature buffers. The existing worker protocol already follows this — reuse the same pattern.

**Step 1: Extend the worker message protocol (`meshBuilder.worker.ts`)**

The existing `buildRoads` and `buildBuildings` message types already do exactly what the export pipeline needs — they build terrain geometry internally for BVH raycasting, then build road/building geometry and return typed arrays. The export pipeline can reuse these exact message types. No new message types are needed.

However, the export pipeline currently passes `terrainGeometry` directly to `buildAllBuildings` and `buildRoadGeometry` for BVH raycasting. The worker already does this internally (builds terrain, then uses it for raycasting). So the export can simply call the same worker functions with the same parameters.

No changes to `meshBuilder.worker.ts` are needed — the existing protocol supports the export use case as-is.

**Step 2: Add export-oriented client functions (`meshBuilderClient.ts`)**

Add two new functions that wrap the existing worker calls but without stale-result rejection (export never has stale results — there's only one active export at a time):

```typescript
/**
 * Build road geometry in the worker for export. Unlike buildRoadsInWorker,
 * this does not use stale-result rejection (export has no concurrent calls).
 */
export function buildRoadsForExport(
  features: RoadFeature[],
  bbox: BoundingBox,
  elevData: ElevationData,
  terrainParams: TerrainMeshParams,
  roadParams: Omit<RoadGeometryParams, 'terrainGeometry'>,
  smoothingLevel = 25
): Promise<MeshArrays | null> {
  const id = nextRequestId++;
  const w = getWorker();

  return new Promise<MeshArrays | null>((resolve, reject) => {
    pending.set(id, { resolve, reject });
    w.postMessage({
      id,
      type: 'buildRoads',
      features,
      bbox,
      elevData: serializeElevData(elevData),
      terrainParams,
      roadParams,
      smoothingLevel,
    });
  });
}

/**
 * Build building geometry in the worker for export.
 */

function meshArraysToGeometry(arrays: MeshArrays): THREE.BufferGeometry {
  const geo = new THREE.BufferGeometry();
  geo.setAttribute('position', new THREE.Float32BufferAttribute(arrays.positions, 3));
  if (arrays.normals) {
    geo.setAttribute('normal', new THREE.Float32BufferAttribute(arrays.normals, 3));
  }
  if (arrays.index) {
    geo.setIndex(new THREE.BufferAttribute(arrays.index, 1));
  }
  if (!arrays.normals) {
    geo.computeVertexNormals();
  }
  return geo;
}
```
