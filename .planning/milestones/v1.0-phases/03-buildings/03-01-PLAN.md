3. Create `src/lib/buildings/overpass.ts`:
   - `fetchBuildingData(bbox: BoundingBox): Promise<unknown>` — POST to `https://overpass-api.de/api/interpreter`
   - Query format: `[out:json][timeout:60][maxsize:33554432][bbox:${sw.lat},${sw.lon},${ne.lat},${ne.lon}]; (way["building"]; way["building:part"]; relation["building"]["type"="multipolygon"];); out geom;`
   - CRITICAL: Overpass bbox order is `south,west,north,east` = `sw.lat,sw.lon,ne.lat,ne.lon` — NOT the GeoJSON order
   - Return the raw JSON response. Throw on non-200 status.

4. Create `src/lib/buildings/parse.ts`:
   - `parseBuildingFeatures(overpassData: unknown): BuildingFeature[]`
   - Use `osmtogeojson(overpassData)` to convert to GeoJSON FeatureCollection
   - For each feature with `geometry.type === 'Polygon'`: extract outer ring (coordinates[0]) and holes (coordinates[1..n])
   - For `MultiPolygon`: extract each polygon separately as individual BuildingFeature entries
   - Extract from properties: `height`, `building:levels`, `roof:shape` (default 'flat'), `roof:height`, `building` (type tag)
   - Skip features with no coordinates or empty rings

5. Create `src/lib/buildings/height.ts`:
   - `resolveHeight(properties: Record<string, string | undefined>, footprintAreaM2?: number): number`
   - Cascade: (1) `parseFloat(properties.height)` if valid > 0, (2) `parseInt(properties['building:levels']) * 3.5` if valid > 0, (3) footprint-area heuristic if footprintAreaM2 is provided: `area < 60 → 5m (shed/garage), area < 200 → 7m (house), area < 600 → 10m (small commercial), area >= 600 → 14m (large building)`, (4) type defaults lookup by `properties.building` value
   - The footprint-area heuristic (tier 3) fulfills BLDG-03's named fallback. Callers compute footprintAreaM2 using the shoelace formula on the UTM-projected outer ring.
   - Type defaults map: house=7, residential=7, apartments=14, commercial=8, industrial=6, church=12, school=9, yes=7 (fallback)
   - Handle unit strings in height tag: `parseFloat("12 m")` = 12, `parseFloat("40'")` needs feet conversion (multiply by 0.3048)
   - `resolveRoofHeight(properties: Record<string, string | undefined>, wallHeight: number): number`
   - If `roof:height` tag exists, parseFloat it. Otherwise default: flat=0, gabled=wallHeight*0.3, hipped=wallHeight*0.3, pyramidal=wallHeight*0.4

This task creates the pure math layer — elevation sampling, polygon triangulation, and wall construction — each with dedicated tests. These modules have no Three.js dependencies and can be validated independently before composition in Task 3.

1. Create `src/lib/buildings/elevationSampler.ts`:
   - `sampleElevationAtLonLat(lon, lat, bbox: BoundingBox, elevData: ElevationData): number`
   - Normalize lon/lat to [0,1] within bbox: `tx = (lon - sw.lon) / (ne.lon - sw.lon)`
   - CRITICAL Y-axis convention (matching terrain.ts): `ty = (ne.lat - lat) / (ne.lat - sw.lat)` — ne.lat is row 0 (north)
   - Compute grid coordinates: `gx = clamp(tx, 0, 1) * (gridSize - 1)`, `gy = clamp(ty, 0, 1) * (gridSize - 1)`
   - Bilinear interpolation of 4 surrounding grid cells
   - Return elevation in meters

2. Create `src/lib/buildings/__tests__/elevationSampler.test.ts`:
   - Test center of bbox returns center grid value
   - Test corners return corner grid values (no interpolation needed)
   - Test midpoint between two grid cells returns average
   - Test points outside bbox are clamped to edge
   - Use a simple 3x3 or 5x5 synthetic grid with known values

3. Create `src/lib/buildings/footprint.ts`:
   - `triangulateFootprint(outer: [number, number][], holes: [number, number][][]): { flatVertices: number[]; indices: number[] }`
   - Flatten outer ring vertices as [x0, y0, x1, y1, ...] (2D)
   - Record hole start indices
   - Call `earcut(flatVertices, holeIndices.length > 0 ? holeIndices : undefined, 2)`
   - Return flatVertices and triangle indices

4. Create `src/lib/buildings/__tests__/footprint.test.ts`:
   - Test simple square produces 2 triangles (6 indices)
   - Test L-shaped polygon produces correct number of triangles
   - Test square with hole produces correct triangulation (courtyard)
   - Verify all returned indices are within bounds of flatVertices

5. Create `src/lib/buildings/walls.ts`:
   - `buildWalls(ringXY: [number, number][], baseZmm: number[], buildingHeightMM: number): Float32Array`
   - For each consecutive pair of ring vertices, create a quad (2 triangles) connecting base to top
   - Each vertex has its own baseZ (per-vertex terrain elevation in mm) — this is how buildings follow slopes (BLDG-04)
   - Top Z = baseZ[i] + buildingHeightMM for vertex i
   - Winding: outward-facing CCW. Use the wall extrusion pattern from 03-RESEARCH.md Pattern 5.
   - IMPORTANT: Ring winding must produce outward normals. After UTM projection, OSM outer rings may be CW. Detect winding with signed area: if signed area > 0 (CW in screen space), reverse the ring before building walls.
   - Return Float32Array of positions (n_segments * 6 vertices * 3 coords)
   - Also export `computeSignedArea(ring: [number, number][]): number` — used by walls.ts internally for winding detection AND by merge.ts to compute footprint area in M² for the height heuristic.

6. Create `src/lib/buildings/__tests__/walls.test.ts`:

   - Test winding detection: verify signed area is negative for CCW ring, positive for CW ring
   - Test wall normals point outward (cross product of two triangle edges should face away from ring centroid)

This task composes the math primitives from Task 2 into building geometry and the full merge pipeline. The critical constraint is zScale consistency with terrain.ts.

1. Create `src/lib/buildings/roof.ts`:
   - `buildFlatRoof(topRingXY: [number, number][], topZmm: number[], holes: [number, number][][] = []): Float32Array`
   - Use `triangulateFootprint()` to get triangle indices for the top cap
   - Each vertex top Z = topZmm[i] (per-vertex, already computed as baseZmm[i] + heightMM upstream)
   - Return Float32Array of positions for the roof cap triangles
   - Also export `buildFloorCap(ringXY, baseZmm, holes)`: same as flat roof but at base elevation with reversed winding (normal points down)

2. Create `src/lib/buildings/buildingMesh.ts`:
   - `buildSingleBuilding(outerRingMM: [number, number][], holesMM: [number, number][][], baseZmm: number[], buildingHeightMM: number, roofShape: string, roofHeightMM: number): THREE.BufferGeometry`
   - Compose: floor cap (buildFloorCap) + walls (buildWalls) + roof (buildFlatRoof for now, gabled/hipped/pyramidal in Plan 02)
   - Merge into single BufferGeometry using `mergeGeometries`
   - This function takes coordinates already in mm local space — coordinate projection happens upstream

3. Create `src/lib/buildings/merge.ts`:
   - `buildAllBuildings(features: BuildingFeature[], bbox: BoundingBox, elevData: ElevationData, params: BuildingGeometryParams): THREE.BufferGeometry`
   - Compute zScale identically to terrain.ts: `horizontalScale = params.widthMM / params.geographicWidthM`, then `zScale = horizontalScale * params.exaggeration`. Apply the same minHeightMM floor logic as terrain.ts if needed (check elevRange).
   - For each BuildingFeature:
     a. Resolve height in meters via `resolveHeight(properties, footprintAreaM2)` — compute footprintAreaM2 from UTM-projected outer ring using `computeSignedArea()` from walls.ts (take absolute value)
     b. For each vertex in outer ring + holes: project [lon,lat] to UTM using `wgs84ToUTM()`, then to local mm space (subtract bbox center UTM, multiply by horizontalScale)
     c. For each vertex: sample terrain elevation via `sampleElevationAtLonLat()`, convert to mm: `baseZmm = (elevationM - params.minElevationM) * zScale` — this mirrors terrain.ts line 124: `z = (elevation - minElevation) * zScale`
     d. Convert building height from meters to mm: `heightMM = heightM * horizontalScale * params.exaggeration`
     e. Call `buildSingleBuilding()` for each building
   - Merge all individual building geometries using `mergeGeometries()`
   - Center offset: bbox center in UTM = projectToUTM(centroidLon, centroidLat, utmZone). Each building vertex UTM coord becomes (utmX - centerX) * horizontalScale for X, (utmY - centerY) * horizontalScale for Y — matching the terrain mesh centering.
   - CRITICAL: The Y-axis convention must match terrain.ts — UTM Y increases north = mesh Y increases north = positive Y. This should be automatic since UTM northing increases north.
   - CRITICAL: zScale and base offset MUST match terrain.ts exactly. The formula is: `baseZmm[i] = (sampleElevationAtLonLat(lon, lat, bbox, elevData) - params.minElevationM) * zScale`. This ensures building Z=0 aligns with terrain Z=0 (both relative to minElevation).
   - Return merged BufferGeometry or null if no buildings found
