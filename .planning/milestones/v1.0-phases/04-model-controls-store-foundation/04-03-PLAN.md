    Wire targetHeightMM into the Z-scale computation so a non-zero value overrides the natural Z scale in both terrain and building geometry.

    **In `src/lib/mesh/terrain.ts`:**

    1. Add `targetHeightMM?: number` to the `TerrainMeshParams` interface (optional, default undefined/0 = auto behavior, no breaking change to existing callers).

    2. In `buildTerrainGeometry`, after computing `elevRange` and `horizontalScale`, add a new zScale branch:
       - If `targetHeightMM > 0 && elevRange > 0`:
         - `terrainReliefMM = targetHeightMM - basePlateThicknessMM` (where `basePlateThicknessMM` is NOT in the params — the caller must subtract it, OR we add it to params. However, basePlateThicknessMM is not currently in `TerrainMeshParams`.)
         - Actually, `targetHeightMM` represents the total Z the user wants (relief + base). The base plate is added at `-basePlateThicknessMM` in `buildSolidMesh`. So the terrain surface max Z must be `targetHeightMM - basePlateThicknessMM`. The existing `zScale` determines terrain Z via `(elevation - minElevation) * zScale`, so the max Z is `elevRange * zScale`. Therefore: `zScale = (targetHeightMM - basePlateThicknessMM) / elevRange`.
         - BUT `basePlateThicknessMM` is not available in `TerrainMeshParams`. Add it? No — it's already passed separately to `buildSolidMesh`. Instead, pass a new field `targetHeightMM` to the params and let the CALLER compute the relief target. **Decision: targetHeightMM in TerrainMeshParams means "target max terrain surface Z in mm" (not including base plate). The caller is responsible for subtracting basePlateThicknessMM before passing.** This keeps the function focused on the terrain surface.
         - RENAMED for clarity in the params: call it `targetReliefMM?: number`. When `targetReliefMM > 0 && elevRange > 0`, set `zScale = targetReliefMM / elevRange`. This overrides the natural `horizontalScale * exaggeration` computation.
         - When `targetReliefMM` is 0 or undefined, keep existing behavior exactly.
       - If `elevRange === 0`, keep `zScale = 0` and `z = minHeightMM` (flat terrain) regardless of override.

    3. Apply the same logic in `updateTerrainElevation` — the zScale computation must match exactly. `updateTerrainElevation` already receives `TerrainMeshParams` so it gets `targetReliefMM` automatically.

    **In `src/lib/buildings/types.ts`:**

    4. Add `targetReliefMM?: number` to the `BuildingGeometryParams` interface. This is how the override reaches the building merge pipeline.

    **In `src/lib/buildings/merge.ts`:**

    5. In `buildAllBuildings`, after the existing zScale computation block (lines 127-145), add a check: if `params.targetReliefMM && params.targetReliefMM > 0 && elevRange > 0`, override `zScale = params.targetReliefMM / elevRange`. This must produce the SAME zScale as terrain.ts for alignment.

    6. Also update building height conversion: currently `heightMM = heightM * horizontalScale * exaggeration`. When targetReliefMM overrides zScale, building heights must also scale proportionally. Replace with: `heightMM = heightM * zScale` (since zScale already includes the correct mm-per-meter-of-elevation factor, and building heights are in meters above terrain).

    Wait — building heights are in meters (real-world), and the conversion is `heightM * horizontalScale * exaggeration` which converts meters-of-real-world-height to mm at the same vertical scale as terrain. With the override: `zScale = targetReliefMM / elevRange`. The building height in mm should be `heightM * zScale` — but `zScale` is in mm per meter of ELEVATION, not mm per meter of building height. Actually both are the same unit (meters of vertical distance → mm). So `heightMM = heightM * zScale` IS correct when targetReliefMM overrides, and `heightM * horizontalScale * exaggeration` is equivalent to `heightM * zScale` when zScale = horizontalScale * exaggeration (the non-override case). So actually, the cleanest fix is to compute heightMM as `heightM * zScale` in ALL cases (not just override), since zScale already encodes the full vertical conversion. The existing code on line 183 (`heightM * horizontalScale * exaggeration`) is algebraically equal to `heightM * zScale` when not in floor mode. When the TERR-03 floor applies (`zScale = minHeightMM / elevRange`), buildings should also use that scale. Currently they DO (via the same if/else on lines 139-144). So replacing lines 183-184 with `heightMM = heightM * zScale; roofHeightMM = roofHeightM * zScale;` is a safe refactor that also handles the override case.

    **CRITICAL:** Do NOT change the existing behavior when `targetReliefMM` is 0/undefined. Verify by tracing through: `targetReliefMM = undefined` → zScale = `horizontalScale * exaggeration` (with minHeightMM floor) → same as before. Correct.
