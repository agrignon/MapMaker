  <action>
    **Fix 1 — MapView.tsx: map instance lookup (UAT tests 4-9)**

    Line 17 currently reads:
    ```
    const mapInstance = maps['main-map']?.getMap() ?? null;
    ```

    Change to:
    ```
    const mapInstance = maps.current?.getMap() ?? null;
    ```

    Reason: Without a `<MapProvider>` ancestor, `useMap()` only populates the `current` key (from MapContext set by the `<Map>` component itself). The id-based lookup `maps['main-map']` requires `MountedMapsContext` which is only provided by `<MapProvider>`. Since this app has a single map and `<MapInteractions>` is already a child of `<Map>`, `maps.current` is the correct and reliable accessor. Do NOT add `<MapProvider>` — it is unnecessary for a single-map app and would add complexity.

    **Fix 2 — MapView.tsx: API key runtime guard (UAT tests 1, 4-9)**

    Replace the silent `?? ''` fallback on line 6:
    ```
    const MAPTILER_KEY = import.meta.env.VITE_MAPTILER_KEY ?? '';
    ```

    With a runtime guard that shows a visible error when the key is missing:
    ```typescript
    const MAPTILER_KEY = import.meta.env.VITE_MAPTILER_KEY as string | undefined;
    ```

    Then in the `MapView` component, before the `<Map>` return, add a guard:
    ```typescript
    if (!MAPTILER_KEY) {
      return (
        <div className="flex items-center justify-center h-full bg-gray-900 text-white p-8">
          <div className="text-center max-w-md">
            <h2 className="text-xl font-bold mb-2">MapTiler API Key Missing</h2>
            <p className="text-gray-400 mb-4">
              Copy <code className="text-amber-400">.env.example</code> to <code className="text-amber-400">.env</code> and set your MapTiler API key.
            </p>
            <code className="block bg-gray-800 p-3 rounded text-sm text-left">
              cp .env.example .env<br/>
              # Edit .env and set VITE_MAPTILER_KEY=your_key
            </code>
          </div>
        </div>
      );
    }
    ```

    Update the MAP_STYLE template literal to use the now-typed MAPTILER_KEY (it will always be a string inside the `<Map>` render path due to the guard above).

    **Fix 3 — SearchOverlay.tsx: API key guard (UAT tests 2-3)**

    Replace line 9:
    ```
    const API_KEY = import.meta.env.VITE_MAPTILER_KEY ?? '';
    ```

    With:
    ```typescript
    const API_KEY = (import.meta.env.VITE_MAPTILER_KEY as string | undefined) ?? '';
    ```

    This is a minor typing improvement. The real guard is in MapView.tsx — if the key is missing, the entire map panel shows the error message, so SearchOverlay never mounts. However, also add a guard in SearchOverlay to not render the GeocodingControl when API_KEY is empty:
    ```typescript
    if (!API_KEY) return null;
    ```

    This prevents the red "Something went wrong" error from ever appearing — if somehow SearchOverlay mounts without a key, it renders nothing instead of a broken geocoding widget.

    **Do NOT create or modify .env** — that is a user action (they must supply their own MapTiler API key). The plan addresses only code fixes.
